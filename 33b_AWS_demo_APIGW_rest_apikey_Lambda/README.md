# Terraform AWS: API Gateway REST API with API Key and Lambda Integration

This Terraform project demonstrates how to secure an **AWS API Gateway REST API (v1)** method using an **API Key and a Usage Plan**. The API Gateway is integrated with an AWS Lambda function that serves as the backend.

This project builds upon the concepts in `33_AWS_demo_APIGW_rest_Lambda` by adding an authentication layer.

## Purpose

The primary goal of this project is to illustrate how to control access to your API Gateway REST API endpoints by requiring clients to provide a valid API key. This is useful for:
*   Authenticating and authorizing client applications.
*   Tracking API usage by key.
*   Implementing throttling and quotas on a per-key basis using Usage Plans.

This project showcases:
*   Configuring an API Gateway method to require an API key.
*   Creating an API Key.
*   Creating a Usage Plan and associating it with the API stage and the API Key.
*   Demonstrating how clients must include the API key in the `x-api-key` header to access the secured endpoint.

## Key Components

1.  **AWS Lambda Function (`aws_lambda_function.demo33b`):**
    *   **Runtime:** Python 3.11 (`python3.11`).
    *   **Handler:** `lambda.lambda_handler` (from `lambda.py`).
    *   **Source Code (`lambda.py`):**
        *   A simple Python function that returns a JSON message: `{"message": "hello world demo33b"}`.
        *   The `lambda.py` script is packaged into a zip file by Terraform.
    *   **IAM Role (`aws_iam_role.demo33b_lambda`):**
        *   An IAM role with the `AWSLambdaBasicExecutionRole` policy attached, allowing the Lambda function to write logs to Amazon CloudWatch Logs.
2.  **API Gateway REST API (`aws_api_gateway_rest_api.demo33b`):**
    *   **Type:** REST API (v1), Regional endpoint.
    *   **Resource (`aws_api_gateway_resource`):** Creates a resource path (e.g., `/${var.apigw_path1}`).
    *   **Method (`aws_api_gateway_method.proxy`):**
        *   Defines a `GET` HTTP method for the resource.
        *   `authorization = "NONE"`: Standard authorization is none, as we are using API keys for this layer of access control.
        *   **`api_key_required = true`**: This is the crucial setting that mandates an API key for invoking this method.
    *   **Integration (`aws_api_gateway_integration`):**
        *   Configured as `AWS_PROXY` to integrate the `GET` method with the Lambda function.
    *   **Deployment and Stage (`aws_api_gateway_deployment`, `aws_api_gateway_stage`):**
        *   Deploys the API to a stage (e.g., `demo33b-stage1`).
3.  **API Key and Usage Plan:**
    *   **API Key (`aws_api_gateway_api_key.demo33b_apikey1`):**
        *   An API key is created. The actual key value is auto-generated by AWS and can be retrieved from Terraform outputs.
        *   `enabled = true`.
    *   **Usage Plan (`aws_api_gateway_usage_plan.demo33b_usage_plan1`):**
        *   A usage plan is defined to manage how the associated API key(s) can be used.
        *   It is associated with the API stage (`aws_api_gateway_stage.demo33b_stage1.stage_name`).
        *   **Note on Throttling/Quotas:** The Terraform configuration includes commented-out examples for `quota_settings` (e.g., limit, period) and `throttle_settings` (e.g., burst_limit, rate_limit). These can be uncommented and configured to control API usage per key. For this demo, they are not actively enforced.
    *   **Usage Plan Key (`aws_api_gateway_usage_plan_key.demo33b_usage_plan_key1`):**
        *   This resource links the created API Key (`aws_api_gateway_api_key.demo33b_apikey1.id`) to the Usage Plan (`aws_api_gateway_usage_plan.demo33b_usage_plan1.id`).
4.  **API Gateway Logging & Lambda Permissions:**
    *   Configured similarly to project `33_AWS_demo_APIGW_rest_Lambda`, with an IAM role for API Gateway logging and Lambda permissions for API Gateway invocation.

## Highlights

*   **API Key Security:** Demonstrates a common method for securing API Gateway endpoints, requiring clients to present a valid API key.
*   **Usage Plans:** Shows the role of usage plans in associating API keys with specific API stages and as a mechanism for managing API access tiers, throttling, and quotas (though throttling/quotas are not actively configured in this example).
*   **`x-api-key` Header:** Emphasizes that clients must include the API key value in the `x-api-key` HTTP header when making requests to the secured endpoint.
*   **Granular Control of REST APIs:** Leverages the detailed configuration capabilities of API Gateway REST APIs (v1).

## Key Configuration Variables

*   `aws_region`: The AWS region where all resources will be deployed (e.g., "us-east-1").
*   `apigw_path1`: The path component for the API Gateway resource (e.g., "greetsecure").

## Usage

1.  **Initialize Terraform:**
    ```bash
    terraform init
    ```
2.  **Plan Changes:**
    Review the resources that Terraform will create.
    ```bash
    terraform plan
    ```
3.  **Apply Changes:**
    Provision the AWS resources.
    ```bash
    terraform apply
    ```
    Confirm by typing `yes`. Terraform will output:
    *   `apigw_invoke_url_on_path1`: The invocation URL for the API Gateway endpoint.
    *   `api_key_value`: The actual value of the generated API key. **This is sensitive information.**

## Verifying the Setup

After successful deployment:

1.  **Get the Invocation URL and API Key Value:**
    Note these values from the `terraform output` after applying.

2.  **Test WITHOUT API Key (Expect 403 Forbidden):**
    Use `curl` (or a similar tool) to make a request to the invocation URL without including the API key.
    ```bash
    curl -v "<apigw_invoke_url_on_path1_output_value>"
    # Example: curl -v https://abcdef123.execute-api.us-east-1.amazonaws.com/demo33b-stage1/greetsecure
    ```
    *   **Expected Output:** You should receive an HTTP `403 Forbidden` response. The response body might include `{"message":"Forbidden"}`. The `-v` flag will show headers, and you might see `x-amzn-ErrorType: ForbiddenException`. This confirms the endpoint is protected and requires an API key.

3.  **Test WITH API Key (Expect 200 OK):**
    Now, make a request including the API key value in the `x-api-key` header.
    ```bash
    API_KEY_VALUE="<api_key_value_output_value>" # Replace with the actual API key value
    curl -v --header "x-api-key: $API_KEY_VALUE" "<apigw_invoke_url_on_path1_output_value>"
    # Example:
    # API_KEY_VALUE="yourActualApiKeyOutputtedByTerraform"
    # curl -v --header "x-api-key: $API_KEY_VALUE" https://abcdef123.execute-api.us-east-1.amazonaws.com/demo33b-stage1/greetsecure
    ```
    *   **Expected Output:** You should receive an HTTP `200 OK` response, and the body should be the JSON from the Lambda function:
        ```json
        {"message": "hello world demo33b"}
        ```
        This confirms that providing a valid API key grants access.

4.  **Check API Gateway Execution Logs (if enabled in Stage settings):**
    *   Navigate to CloudWatch Logs. The log group (e.g., `/aws/apigateway/demo33b_rest_api_access_logs`) would show details of requests, including whether they were allowed or denied.

5.  **Check Lambda Execution Logs:**
    *   Navigate to CloudWatch Logs for the Lambda function (`/aws/lambda/demo33b_lambda_py`). You should only see invocation logs for successful requests (those made with the valid API key).

This testing procedure validates that the API key security is correctly implemented and enforced by API Gateway.
