# Terraform AWS: RDS Aurora MySQL Cluster (Provisioned) with EC2 Client

This Terraform project provisions an AWS environment featuring an AWS RDS Aurora MySQL cluster (provisioned capacity) with multiple database instances spread across different Availability Zones. It also includes a pre-configured Amazon Linux 2 EC2 instance to act as a client for the Aurora cluster, along with helper scripts for checking Aurora MySQL engine versions.

## Key Features & Concepts

*   **AWS RDS Aurora MySQL Cluster:** Deploys a highly available and scalable Aurora MySQL cluster. Aurora is MySQL-compatible and offers enhanced performance and availability over standard RDS MySQL.
*   **Multi-Instance Cluster:** The cluster is provisioned with multiple `aws_rds_cluster_instance` resources (e.g., one writer and multiple readers), distributed across different Availability Zones for high availability and read scaling.
*   **Cluster Endpoints:** Clients connect to the Aurora cluster using cluster endpoints:
    *   **Cluster Endpoint (Writer):** Directs traffic to the primary writer instance.
    *   **Reader Endpoint:** Distributes read traffic among the available Aurora Replicas.
*   **Private Accessibility:** Aurora cluster instances are configured with `publicly_accessible = false`, restricting direct access from the public internet.
*   **DB Subnet Group:** The Aurora cluster is placed within an `aws_db_subnet_group`. This project includes multiple private subnets for the Aurora instances and, somewhat unconventionally, also includes the public client subnet in the DB subnet group. This might be for specific routing or accessibility tests but usually, DB subnet groups contain only private subnets.
*   **EC2 Client with Cloud-Init:** An Amazon Linux 2 EC2 instance is launched and configured using a cloud-init script. This script is templated with the Aurora cluster's endpoint and master username for easy MySQL client setup.
*   **Engine Version Helper Scripts:** Includes shell scripts (`list_versions_v1.sh`, `list_versions_v2_v3.sh`) to help list available Aurora MySQL engine versions, useful for selecting an appropriate version during configuration.
*   **Security Groups for Controlled Access:** Dedicated security groups for the Aurora cluster and EC2 manage traffic flow.

## AWS Resources Provisioned

*   **VPC (Virtual Private Cloud):**
    *   A new VPC with an associated Internet Gateway (IGW).
*   **Subnets:**
    *   **Public Client Subnet (`var.cidr_subnet_public`):** A single public subnet to host the EC2 client instance.
    *   **Private RDS Subnets (`var.cidrs_subnet_private_rds`):** Multiple private subnets (e.g., three) in different Availability Zones (`var.az_rds_list`) to host the Aurora cluster instances.
*   **AWS RDS Aurora MySQL Cluster (`aws_rds_cluster`):**
    *   Engine set to `aurora-mysql`.
    *   Configurable options: `aurora_mysql_engine_version`, `aurora_mysql_master_username`, `aurora_mysql_db_name`.
    *   A random master password is generated by RDS.
    *   Deployed across the Availability Zones specified in `var.az_rds_list`.
    *   **DB Subnet Group (`aws_db_subnet_group`):** Created from the private RDS subnets and the public client subnet.
    *   Associated with a dedicated security group (`aws_security_group.demo13_rds`).
*   **Aurora Cluster Instances (`aws_rds_cluster_instance`):**
    *   Multiple instances (e.g., 3, controlled by `var.aurora_mysql_nb_instances`) are provisioned and attached to the cluster. This typically results in one writer instance and multiple reader instances.
    *   Configurable instance class (`var.aurora_mysql_instance_class`).
    *   `publicly_accessible = false`.
*   **RDS Security Group (`demo13-rds-sg`):**
    *   Allows inbound TCP port 3306 (MySQL) from the VPC's CIDR block (`var.cidr_vpc`), enabling access from the EC2 client.
*   **Linux EC2 Client Instance:**
    *   An Amazon Linux 2 instance (type `var.al2_inst_type`) launched in the public client subnet.
    *   An **Elastic IP (EIP)** is associated for a static public IP address.
    *   Uses a **cloud-init script** (from `var.al2_cloud_init_script` template) populated with the Aurora cluster endpoint and master username. The script installs MySQL client tools.
    *   Associated with its own security group (`aws_default_security_group.demo13_ec2`, typically the VPC's default SG modified or a new one).
*   **EC2 Client Security Group (`demo13-ec2-sg` - typically VPC's default SG):**
    *   Allows inbound SSH (TCP port 22) from `authorized_ips`.
    *   Allows all outbound traffic (or at least traffic to the RDS security group on port 3306).
*   **Network ACLs (NACLs):**
    *   Configured for public and private subnets to allow necessary traffic (SSH, MySQL within VPC, internet access for EC2).

## Architecture

```
        [ AWS Cloud - Region: var.aws_region ]
                               |
        +-----------------------------------------------------------------+
        |                             VPC                                 |
        |                       (var.cidr_vpc)                            |
        |                                                                 |
        |  +-----------------------+  +---------------------------------+ |
        |  | Public Client Subnet  |  | Private RDS Subnets (Multi-AZ)  | | DB Subnet Group
        |  | (var.cidr_subnet_public)|  | (var.cidrs_subnet_private_rds)  |<-- (Spans Public & Private)
        |  |-----------------------|  |---------------------------------| |
        |  | EC2 Client Instance   |  | [Aurora Cluster Endpoint]       | |
        |  | (EIP, AL2)            |<---> Writer Instance (MySQL 3306) | |
        |  | - MySQL Client        |  |   Instance 1 (AZ1)              | |
        |  | (SG: demo13-ec2)      |  |   Instance 2 (AZ2) (Reader)   | |
        |  |        |              |  |   Instance 3 (AZ3) (Reader)   | |
        |  +--------|--------------+  | (SG: demo13-rds)                | |
        |           | (SSH)           +---------------------------------+ |
        |           ▼                                                     |
        |     [Internet Gateway]                                          |
        +-----------------------------------------------------------------+
                      (Internet)

Traffic Flow:
  - User SSH -> EC2 Client Instance (via IGW & EIP).
  - EC2 Client -> Aurora Cluster Endpoint (Private IP, within VPC, TCP 3306).
    - The cluster endpoint directs write traffic to the writer instance and read traffic to reader instances (if using reader endpoint).
```
The Aurora cluster instances are in private subnets. The EC2 client is in a public subnet for SSH access and communicates with the Aurora cluster using its cluster endpoint(s) via private IP addresses.

## Key Configuration Variables

*   **General AWS:**
    *   `aws_region`: AWS region (e.g., "us-east-1").
    *   `az_rds_list`: List of Availability Zones for Aurora instances (e.g., `["us-east-1a", "us-east-1b", "us-east-1c"]`).
    *   `cidr_vpc`: CIDR block for the VPC (e.g., "10.100.0.0/16").
    *   `cidr_subnet_public`: CIDR for the public client subnet.
    *   `cidrs_subnet_private_rds`: List of CIDRs for the private RDS subnets.
    *   `authorized_ips`: IPs/CIDRs for SSH access to the EC2 client.
*   **Aurora MySQL Specific:**
    *   `aurora_mysql_cluster_identifier`: Cluster identifier (e.g., "demo13-aurora-mysql").
    *   `aurora_mysql_engine_version`: Aurora MySQL engine version (e.g., "8.0.mysql_aurora.3.02.0").
    *   `aurora_mysql_master_username`: Master username (e.g., "admin").
    *   `aurora_mysql_db_name`: Initial database name (e.g., "demodb").
    *   `aurora_mysql_instance_class`: Instance class for Aurora instances (e.g., "db.r6g.large").
    *   `aurora_mysql_nb_instances`: Number of instances in the cluster (e.g., 3).
*   **EC2 Client Specific:**
    *   `al2_inst_type`: EC2 instance type (e.g., "t3.micro").
    *   `al2_ssh_key_name`: Name of an existing EC2 Key Pair for SSH.
    *   `al2_cloud_init_script`: Path to the cloud-init template file.

## Connecting to Aurora

After successful deployment:

1.  **SSH into the EC2 Client Instance:**
    Use its Elastic IP (EIP) and your SSH key.
    ```bash
    ssh -i /path/to/your/ssh-key.pem ec2-user@<EIP_EC2_Instance>
    ```

2.  **Connect to Aurora MySQL using the Cluster Endpoint:**
    The cloud-init script (`var.al2_cloud_init_script`) should have installed MySQL client tools. The Aurora cluster details (cluster endpoint, master username) are passed as template variables to this script.
    Connect using the **cluster endpoint** (for writes) or the **reader endpoint** (for reads):
    ```bash
    # To connect to the writer instance (for read/write operations)
    mysql -h <aurora_cluster_endpoint_address> -u <aurora_mysql_master_username> -p <aurora_mysql_db_name>

    # To connect to reader instances (for read-only operations, if reader endpoint is different and known)
    # mysql -h <aurora_cluster_reader_endpoint_address> -u <aurora_mysql_master_username> -p <aurora_mysql_db_name>
    ```
    *   Enter the **randomly generated master password** when prompted. You'll need to retrieve this (e.g., from AWS Secrets Manager if integrated, or from Terraform output if exposed - not recommended for production).
    *   `<aurora_cluster_endpoint_address>`: The **cluster endpoint DNS name** of the Aurora cluster (available in Terraform outputs or AWS console).
    *   `<aurora_cluster_reader_endpoint_address>`: The **reader endpoint DNS name** (if different and you wish to target readers specifically).
    *   `<aurora_mysql_master_username>`: The master username (e.g., "admin").
    *   `<aurora_mysql_db_name>`: The database name.

    Using the cluster endpoint is crucial as it abstracts the individual instances and allows Aurora to manage connections, especially during failovers or scaling.

## Helper Scripts for Engine Versions

This project includes two shell scripts in its root directory:
*   `list_versions_v1.sh`
*   `list_versions_v2_v3.sh`

**Purpose:** These scripts are designed to query AWS for available Aurora MySQL engine versions using the AWS CLI. They can be helpful when you need to decide which `aurora_mysql_engine_version` to specify in your Terraform configuration.
*   `list_versions_v1.sh`: Likely targets older Aurora MySQL versions (e.g., compatible with MySQL 5.6, if "v1" refers to Aurora MySQL version 1).
*   `list_versions_v2_v3.sh`: Likely targets newer Aurora MySQL versions (e.g., compatible with MySQL 5.7 or 8.0, if "v2/v3" refers to Aurora MySQL versions 2 or 3).

**How to Use:**
1.  Ensure your AWS CLI is configured with appropriate permissions (e.g., `rds:DescribeDBEngineVersions`).
2.  Run these scripts from your local machine where you have the AWS CLI and these scripts.
    ```bash
    ./list_versions_v1.sh
    ./list_versions_v2_v3.sh
    ```
    The output will list available engine versions that you can then use for the `aurora_mysql_engine_version` variable in your Terraform setup. These scripts are **not** copied to the EC2 instance by default but are tools for the Terraform operator.

This setup provides a robust, highly available Aurora MySQL cluster suitable for various applications, accessible from a pre-configured EC2 client. Remember to manage the master password securely.
