# Terraform AWS: Provisioned Redshift Cluster (Single-Node) in Private Subnet

This Terraform project demonstrates how to provision an **AWS Redshift cluster** (single-node, provisioned capacity) within a private subnet of a new VPC. It also includes considerations and variables for an optional EC2 client instance that can be used to connect to the Redshift cluster.

## Purpose

The primary goal of this project is to illustrate:
1.  The deployment of a single-node provisioned AWS Redshift cluster.
2.  Placement of the Redshift cluster in a private subnet for enhanced security.
3.  Configuration of network components (VPC, subnets, NAT Gateway, Security Groups) to allow controlled access to the Redshift cluster.
4.  The setup includes variables and a structure for an optional EC2 client instance, though its provisioning is managed separately (e.g., located in an `off/` subdirectory).

This setup is suitable for development, testing, or small-scale data warehousing scenarios where a single-node Redshift cluster is sufficient.

## Key Components

1.  **VPC Infrastructure:**
    *   A new VPC configured with:
        *   A **public subnet**: Hosts a NAT Gateway and potentially the optional EC2 client instance.
        *   A **private subnet**: Where the Redshift cluster is deployed.
    *   An Internet Gateway (IGW) attached to the VPC for the public subnet.
2.  **NAT Gateway:**
    *   Deployed in the public subnet with an Elastic IP.
    *   Provides outbound internet connectivity for resources in the private subnet, such as the Redshift cluster if it needs to access other AWS services that are not available via VPC endpoints (e.g., for loading data from S3 without S3 VPC Gateway Endpoint, or for software updates if custom UDFs were allowed).
3.  **AWS Redshift Cluster (`aws_redshift_cluster.demo36`):**
    *   **Cluster Type:** `single-node`. This provisions a cluster with only one compute node.
    *   **Node Type:** Hardcoded in this example (e.g., to `dc2.large`). This would typically be a variable.
    *   **Database Configuration:**
        *   Configurable database name (`var.redshift_database_name`).
        *   Master username is hardcoded to "awsuser" in this example (`var.redshift_master_username` might be used if not hardcoded in resource).
        *   A **random master password** is generated by Terraform using `random_password`.
    *   **Redshift Subnet Group (`aws_redshift_subnet_group.demo36_sng`):**
        *   Created using the **private subnet**. The Redshift cluster will be launched within this subnet.
    *   **Redshift Security Group (`aws_security_group.demo36_sg_redshift`):**
        *   A dedicated security group for the Redshift cluster.
        *   Allows inbound traffic on the Redshift port (default 5439, configurable via `var.redshift_port`) from the VPC's public subnet CIDR block. This enables clients in the public subnet (like the optional EC2 instance) to connect.
    *   **`enhanced_vpc_routing = true`**: This feature forces all COPY and UNLOAD traffic between your cluster and your data repositories (S3, DynamoDB, EMR, EC2) through your VPC. If false, Redshift routes traffic through the internet.
    *   `publicly_accessible = false`: The cluster is not directly accessible from the internet.
4.  **(Optional) EC2 Client Instance:**
    *   The project structure includes an `off/06_ec2_instance_db_client.tf` file, which suggests that an EC2 client instance can be optionally provisioned.
    *   Variables for configuring this EC2 client (e.g., `al2_inst_type`, `al2_ssh_key_name`) are present in the main `01_variables.tf` file.
    *   If provisioned, this client would typically reside in the **public subnet** and be used to connect to the Redshift cluster using SQL client tools.

## Network Configuration

*   **Redshift in Private Subnet:** The Redshift cluster is deployed into a private subnet, meaning it does not have a direct route to the internet via an Internet Gateway. This enhances security by isolating the database.
*   **NAT Gateway for Outbound:** The private subnet has a route to the NAT Gateway (located in the public subnet) for any outbound internet access the Redshift cluster might need (e.g., accessing AWS services that don't have VPC endpoints configured, or for specific Redshift commands that might interact with external resources).
*   **Controlled Inbound Access:** The Redshift cluster's security group (`aws_security_group.demo36_sg_redshift`) is configured to allow inbound connections on the Redshift port (default 5439) only from the public subnet's CIDR range. This means only resources within that public subnet (like the optional EC2 client or a bastion host) can directly connect to the Redshift cluster.
*   **Enhanced VPC Routing:** When enabled, this forces COPY and UNLOAD traffic through the VPC, allowing you to use VPC features like security groups and network ACLs to control this traffic, and VPC endpoints to access services like S3 privately.

## Highlights

*   **Single-Node Redshift:** Demonstrates the setup of a cost-effective single-node Redshift cluster suitable for smaller workloads or development.
*   **Private Subnet Deployment:** Emphasizes security by placing the Redshift cluster in a private subnet.
*   **Controlled Network Access:** Shows how security groups are used to restrict access to the Redshift cluster, primarily from resources within the same VPC's public subnet.
*   **Optional EC2 Client:** The project structure accounts for an optional EC2 client, providing flexibility for different connection testing scenarios.

## Key Configuration Variables

*   **General AWS & VPC:**
    *   `aws_region`: AWS region for deployment.
    *   `az`: Primary Availability Zone.
    *   `cidr_vpc`, `cidr_subnet_public`, `cidr_subnet_private`: CIDR blocks.
    *   `authorized_ips`: IPs/CIDRs for SSH access to the optional EC2 client/bastion.
*   **Redshift Specific:**
    *   `redshift_cluster_identifier`: Identifier for the Redshift cluster.
    *   `redshift_database_name`: Name of the initial database.
    *   `redshift_port`: Port for Redshift (default 5439).
    *   *(Note: `redshift_node_type` (e.g., "dc2.large") and `redshift_master_username` (e.g., "awsuser") are hardcoded in the `aws_redshift_cluster` resource in this specific example, but are often variables.)*
*   **(Optional) EC2 Client Specific:**
    *   `al2_inst_type`: Instance type for the optional EC2 client.
    *   `al2_ssh_key_name`: Name of an existing EC2 Key Pair for the client.

## Usage

1.  **Initialize Terraform:**
    ```bash
    terraform init
    ```
2.  **(Optional) Include EC2 Client:**
    If you wish to provision the EC2 client instance, you would typically move or symlink the `off/06_ec2_instance_db_client.tf` file into the root of the project directory before running `plan` or `apply`. (Alternatively, more advanced configurations might use modules or conditional resource creation, but for this structure, file movement is implied for enabling it).
3.  **Plan Changes:**
    Review the resources that Terraform will create.
    ```bash
    terraform plan
    ```
4.  **Apply Changes:**
    Provision the AWS resources. Redshift cluster provisioning can take a significant amount of time (10-15 minutes or more).
    ```bash
    terraform apply
    ```
    Confirm by typing `yes`. Terraform will output the Redshift cluster endpoint, database name, master username, and the **randomly generated master password**.

## Connecting to Redshift

After successful deployment and the Redshift cluster status is "available":

1.  **Retrieve Connection Details:**
    *   **Endpoint:** Get the `redshift_cluster_endpoint` from Terraform output.
    *   **Port:** `var.redshift_port` (default 5439).
    *   **Database Name:** `var.redshift_database_name`.
    *   **Master Username:** "awsuser" (as hardcoded in this example).
    *   **Master Password:** Get the `redshift_master_password` from Terraform output.

2.  **Connection Method:**
    *   **From the Optional EC2 Client (if provisioned in the public subnet):**
        1.  SSH into the EC2 client instance.
        2.  Install a SQL client compatible with PostgreSQL/Redshift (e.g., `psql` or SQL Workbench/J with Redshift JDBC drivers).
            *   For `psql` on Amazon Linux 2: `sudo yum install postgresql`
        3.  Use the SQL client to connect to the Redshift cluster endpoint using the retrieved credentials.
            ```bash
            psql -h <redshift_cluster_endpoint> -p <redshift_port> -U awsuser -d <redshift_database_name>
            # You will be prompted for the password.
            ```
    *   **From a Local Machine (More Complex Setup):**
        *   This setup's security group by default only allows connections from the public subnet's CIDR.
        *   To connect from your local machine, you would need to:
            1.  Modify the Redshift security group (`aws_security_group.demo36_sg_redshift`) to allow inbound traffic on the Redshift port from your specific public IP address.
            2.  Ensure the Redshift cluster's `publicly_accessible` attribute is `true` (which is not the default in this project, it's `false`). If it's `false`, you can't connect directly from the internet even if the SG allows it. You'd need a bastion host or VPN into the VPC.
            *Given the current setup (`publicly_accessible = false` and SG allowing from public subnet), direct connection from a local machine over the internet is not intended without modifications.*

Once connected, you can run SQL queries against your Redshift cluster. Remember that the master password should be managed securely, especially in production (e.g., using AWS Secrets Manager).
