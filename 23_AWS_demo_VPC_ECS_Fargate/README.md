# Terraform AWS: ECS Fargate Service with ALB and FIS Experiment

This Terraform project demonstrates how to provision an **AWS ECS (Elastic Container Service) Fargate service** running Nginx containers. The service is fronted by an **Application Load Balancer (ALB)** for traffic distribution. Additionally, it includes an **AWS Fault Injection Simulator (FIS) experiment template** designed to test the resilience of the ECS service by stopping tasks in a specific Availability Zone.

## Purpose

The primary goals of this project are:
1.  To illustrate the deployment of a containerized application (Nginx) using AWS ECS Fargate, highlighting its serverless nature for running containers without managing underlying EC2 instances.
2.  To show how an Application Load Balancer can be used to expose the Fargate service to users.
3.  To introduce AWS Fault Injection Simulator (FIS) as a tool for chaos engineering, allowing you to test the resilience and fault tolerance of your ECS application.
4.  To provide a suite of helper scripts for interacting with and testing the deployed ECS service and FIS experiments.

## Key Components

### 1. VPC Infrastructure
*   A Virtual Private Cloud (VPC) is set up with public subnets. The ECS Fargate tasks, in this configuration, are launched in these public subnets and assigned public IP addresses to allow them to pull container images from public registries like Docker Hub. (Note: For production, tasks are often placed in private subnets with a NAT Gateway for outbound internet and use VPC endpoints for ECR).
*   An Internet Gateway (IGW) is attached to the VPC.

### 2. Application Load Balancer (ALB) (`04_elb_for_ecs_service.tf`)
*   **`aws_lb`**: A public-facing Application Load Balancer is deployed across the public subnets.
*   **`aws_lb_target_group`**: An IP-based target group is created. This is suitable for Fargate tasks, as Fargate assigns an ENI (and thus an IP address) to each task. The ALB routes traffic to the IP addresses of the healthy Fargate tasks.
*   **`aws_lb_listener`**: An HTTP listener on port 80 is configured to forward requests to the target group.

### 3. ECS Fargate Service (`05_ecs_fargate.tf`)
*   **ECS Cluster (`aws_ecs_cluster.demo23_cluster`):**
    *   A new ECS cluster is created.
    *   Configured with `FARGATE` and `FARGATE_SPOT` default capacity provider strategies.
*   **ECS Task Definition (`aws_ecs_task_definition.demo23_task_def_nginx`):**
    *   Defines the container(s) to be run.
    *   Uses a standard **Nginx** image (e.g., `nginx:latest`).
    *   Specifies CPU and memory allocation for the task (e.g., `cpu = 256`, `memory = 512`).
    *   Defines port mappings (e.g., container port 80 mapped to host port 80 on the task's ENI).
    *   Requires Fargate compatibility (`requires_compatibilities = ["FARGATE"]`).
*   **ECS Service (`aws_ecs_service.demo23_service_nginx`):**
    *   Runs and maintains a specified number of instances of the task definition.
    *   `launch_type = "FARGATE"`: Specifies that tasks will run on AWS Fargate.
    *   `desired_count = 3` (example): Runs three instances of the Nginx task.
    *   **Network Configuration:**
        *   Tasks are launched in the public subnets.
        *   `assign_public_ip = true`: Each Fargate task gets a public IP, primarily for pulling images if not using a private ECR with VPC endpoints.
    *   **Load Balancing:** Integrated with the ALB's target group to distribute traffic to the tasks.
    *   **Deployment Circuit Breaker:** Configured with `enable_ecs_managed_tags = true` and potentially a deployment controller to manage rollbacks if deployments fail (though specific circuit breaker config isn't explicitly detailed in the prompt, it's a common feature).

### 4. AWS Fault Injection Simulator (FIS) (`06_fis_experiment_template.tf`)
*   **IAM Role for FIS (`aws_iam_role.demo23_fis`):**
    *   A dedicated IAM role is created for FIS.
    *   Permissions:
        *   Allows FIS to stop ECS tasks (`ecs:StopTask`).
        *   Allows FIS to write logs to CloudWatch Logs (`logs:CreateLogDelivery`, `logs:PutLogEvents`, etc.).
*   **CloudWatch Log Group (`aws_cloudwatch_log_group.demo23_fis`):**
    *   Created to store logs generated by FIS experiments.
*   **FIS Experiment Template (`aws_fis_experiment_template.demo23_fis_stop_tasks_in_one_az`):**
    *   Defines a reusable template for fault injection experiments.
    *   **Target:**
        *   Identifies ECS tasks to target based on:
            *   ECS Cluster name.
            *   ECS Service name.
            *   A specific Availability Zone (`var.az_for_fis`). This means the experiment will only affect tasks running in that particular AZ.
    *   **Action:**
        *   `aws:ecs:stop-task`: The action to perform on the targets, which is to stop the ECS tasks.
    *   **Stop Condition:** (Not explicitly detailed in prompt, but common) Could be a duration or a CloudWatch alarm. This demo might run for a default duration or indefinitely until manually stopped.
    *   **Role ARN:** Uses the `aws_iam_role.demo23_fis.arn`.

## Highlights

*   **Serverless Container Orchestration:** Demonstrates the ease of deploying and running containerized applications with AWS ECS Fargate, abstracting away server management.
*   **Resilience Testing with FIS:** Shows how AWS Fault Injection Simulator can be used to proactively test the fault tolerance of an application by simulating failures (e.g., stopping tasks in an AZ) and observing how the system responds (e.g., ECS service scheduler replacing tasks, ALB health checks).
*   **Operational Tooling:** The provided helper scripts in the `scripts/` directory facilitate interaction, monitoring, and testing of the ECS environment.

*   **FIS Specific:**
    *   `az_for_fis`: The Availability Zone targeted by the FIS experiment template (e.g., "us-east-1a").

    ```bash

    ```

    Review the resources that Terraform will create.

    Provision the AWS resources.
    ```bash
    terraform apply
    ```
    Confirm by typing `yes`.

## Running FIS Experiments

After successful deployment:

1.  **Navigate to AWS Fault Injection Simulator (FIS) Console:**
    *   Open the AWS Management Console and go to FIS.
    *   Ensure you are in the same AWS region where you deployed the resources.

2.  **Find the Experiment Template:**
    *   In the FIS console, go to "Experiment templates".
    *   You should see the template created by Terraform (e.g., named based on `aws_fis_experiment_template.demo23_fis_stop_tasks_in_one_az`).

3.  **Start an Experiment:**
    *   Select the template.
    *   Click "Start experiment".
    *   You may need to provide a name or tags for the experiment instance.
    *   Confirm and start the experiment.

4.  **Observe the Impact:**
    *   **ECS Console:** Monitor the ECS service. You should see tasks in the targeted AZ being stopped and then replaced by the ECS service scheduler (if the desired count is maintained).
    *   **ALB Target Group:** Check the health of targets in the ALB target group. Instances in the affected AZ might become unhealthy temporarily.
    *   **Application Accessibility:** Access the application via the ALB's DNS name. You should observe if the application remains available, potentially with reduced capacity or slightly increased latency during the task cycling.
    *   **FIS Console:** Monitor the experiment's progress and status.
    *   **CloudWatch Logs:** Check the CloudWatch Log Group created for FIS for logs related to the experiment execution.

This process helps you understand how your ECS Fargate service behaves under specific failure conditions and verify its resilience mechanisms.

## Helper Scripts (`scripts/` directory)

This project includes a `scripts/` directory containing various shell (`.sh`) and Python (`.py`) scripts. These scripts are provided as utilities to:
*   Interact with the deployed ECS service (e.g., list tasks, describe services).
*   Check the status of the web application through the ALB.
*   Potentially trigger or verify the effects of FIS experiments (e.g., by repeatedly checking web access while an experiment is running).
*   Perform other operational tasks like killing specific tasks for ad-hoc testing.

Refer to the individual scripts within the `scripts/` directory for their specific usage and purpose. They can be valuable tools for managing, testing, and understanding the behavior of your ECS Fargate environment.
