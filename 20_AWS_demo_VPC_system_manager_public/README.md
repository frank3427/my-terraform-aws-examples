# Terraform AWS: EC2 Instances Managed by AWS Systems Manager (Public Subnet)

This Terraform project demonstrates how to provision one or more Linux EC2 instances (Amazon Linux 2) in a public subnet, configured to be managed by **AWS Systems Manager (SSM)**.

## Purpose

The primary goal of this project is to showcase the setup required for EC2 instances to be discoverable and manageable through AWS Systems Manager. SSM provides a unified interface for various operational tasks, including:
*   **Run Command:** Execute commands remotely on your instances without needing direct SSH access for every operation.
*   **Session Manager:** Provides secure shell access (or non-interactive commands) to instances without needing to open inbound SSH ports in security groups (though this demo configures SSH access as well). Ideal for enhanced security.
*   **Patch Manager:** Automate patching of your instances.
*   **State Manager:** Define and maintain consistent configurations on your instances.
*   **Inventory:** Collect metadata about your instances and their configurations.

This project focuses on the foundational setup for making instances SSM-manageable.

## Key Components

1.  **VPC Infrastructure:**
    *   A standard VPC with a public subnet and an Internet Gateway (IGW). This allows instances to have direct internet access, which is necessary for the SSM agent to communicate with public SSM service endpoints.
2.  **EC2 Instances:**
    *   A configurable number (`var.nb_instances_linux`) of Amazon Linux 2 EC2 instances.
    *   Each instance is launched in the public subnet and associated with an Elastic IP (EIP) for a static public IP address.
3.  **IAM Role for Systems Manager (`aws_iam_role.demo20_ssm_for_ec2`):**
    *   An IAM role is created specifically for the EC2 instances.
    *   **Policy Attachments:**
        *   `AmazonSSMManagedInstanceCore`: This is the primary AWS managed policy that grants the core permissions required for an instance to be managed by Systems Manager, including communication with the SSM service.
        *   `AmazonEC2RoleforSSM`: An older policy, often included for backward compatibility or specific scenarios, also granting permissions for SSM. (Note: `AmazonSSMManagedInstanceCore` is generally preferred and sufficient for most new setups).
    *   **Instance Profile:** An `aws_iam_instance_profile` is created and associated with this role. This profile is then attached to the EC2 instances, allowing the SSM agent running on them to inherit these permissions.
4.  **SSM Agent:**
    *   Amazon Linux 2 AMIs come with the **SSM agent pre-installed and enabled by default**. Therefore, the cloud-init script in this project does not need to explicitly install or start it.
    *   Once an instance is launched with the appropriate IAM role (providing `AmazonSSMManagedInstanceCore` permissions), the SSM agent automatically registers the instance with the Systems Manager service in that AWS region. The instance should then appear in the "Managed Instances" list in the SSM console.
5.  **SSH Access & `sshcfg` File:**
    *   Standard SSH access to the instances is also configured using EC2 Key Pairs.
    *   A local `sshcfg` file is generated by Terraform using the `templates/sshcfg.template`. This file contains SSH aliases for each provisioned instance, making it convenient to SSH into them using their assigned names (e.g., `ssh demo20-linux-1`).
6.  **Public Accessibility:**
    *   Instances are placed in a public subnet and have direct internet connectivity via the IGW. This allows the SSM agent on the instances to communicate with the public AWS Systems Manager service endpoints without requiring VPC endpoints for SSM (though VPC endpoints are a best practice for private subnets).

## Highlights

*   **Effortless SSM Integration:** Demonstrates how instances become SSM-manageable primarily through the correct IAM role (`AmazonSSMManagedInstanceCore`) when using AMIs with a pre-installed SSM agent (like Amazon Linux 2).
*   **Operational Capabilities via SSM:** Briefly touches upon the types of operations possible once instances are managed by SSM (Run Command, Session Manager, Patch Manager, etc.).
*   **Convenient SSH Access:** The generated `sshcfg` file simplifies direct SSH connections if needed for tasks not covered by SSM or for initial troubleshooting.

## Key Configuration Variables

*   `aws_region`: The AWS region for deploying all resources (e.g., "us-east-1").
*   `az`: The Availability Zone for the public subnet and EC2 instances (e.g., "us-east-1a").
*   `cidr_vpc`: CIDR block for the VPC (e.g., "10.140.0.0/16").
*   `cidr_subnet1`: CIDR block for the public subnet (e.g., "10.140.1.0/24").
*   `authorized_ips`: List of IPs/CIDRs for SSH access to the EC2 instances (e.g., `["YOUR_PUBLIC_IP/32"]`).
*   `inst_type`: EC2 instance type (e.g., "t2.micro").
*   `al2_ssh_key_name`: Name of an existing EC2 Key Pair in the specified region for SSH access.
*   `nb_instances_linux`: The number of Linux EC2 instances to provision (e.g., 2).

## Usage

1.  **Initialize Terraform:**
    ```bash
    terraform init
    ```
2.  **Plan Changes:**
    Review the resources that Terraform will create.
    ```bash
    terraform plan
    ```
3.  **Apply Changes:**
    Provision the AWS resources.
    ```bash
    terraform apply
    ```
    Confirm by typing `yes`. A `sshcfg` file will be generated in your project directory.

## Verifying SSM Management

After successful deployment (allow a few minutes for instances to boot and the SSM agent to register):

1.  **Check Managed Instances in AWS Systems Manager Console:**
    *   Open the AWS Management Console and navigate to "Systems Manager".
    *   In the Systems Manager navigation pane, under "Node Management", click on "Managed instances".
    *   Ensure you are in the same AWS region where you deployed the resources.
    *   Your newly launched EC2 instances should appear in the list with a "Ping status" of "Online". This indicates the SSM agent is running and communicating with the SSM service.

2.  **Test with Run Command:**
    *   In the Systems Manager console, under "Node Management", click on "Run Command".
    *   Click "Run command".
    *   Search for and select the command document `AWS-RunShellScript`.
    *   **Command parameters:**
        *   In the "Commands" box, enter a simple shell command like:
            ```bash
            ifconfig
            # or
            ls -la /home/ec2-user/
            ```
    *   **Targets:**
        *   Choose "Specify instance tags" or "Choose instances manually" and select the EC2 instances provisioned by this project (e.g., based on a tag like "Name" or their instance IDs).
    *   Leave other options as default and click "Run".
    *   After a short while, the command should complete. You can click on the command ID and then view the output for each instance to see the results of your shell command.

3.  **Test with Session Manager (Optional):**
    *   In "Managed instances", select one of your instances.
    *   Click the "Start session" button. This should open a shell session to your instance directly in the browser, without needing an SSH client or open SSH ports in the instance's security group (if you were to restrict SSH access).

Successful completion of these steps confirms that your EC2 instances are correctly configured and managed by AWS Systems Manager. You can now explore other SSM capabilities like Patch Manager, State Manager, and Inventory. Remember to use the generated `sshcfg` file for direct SSH access if needed.
