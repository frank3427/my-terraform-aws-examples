# Terraform AWS: DocumentDB Cluster with EC2 Client

This Terraform project provisions an AWS environment featuring an **AWS DocumentDB (with MongoDB compatibility)** cluster. The cluster is configured with a configurable number of instances and is deployed across multiple Availability Zones using a DB Subnet Group. An Amazon Linux 2 EC2 instance is also provisioned and pre-configured to act as a client for interacting with the DocumentDB cluster using the mongo shell.

## Key Features & Concepts

*   **AWS DocumentDB:** A fast, scalable, highly available, and fully managed document database service that supports MongoDB workloads.
*   **Multi-AZ Deployment:** The DocumentDB cluster instances are implicitly distributed across Availability Zones covered by the DB Subnet Group. While individual instances are in specific AZs, the cluster endpoint provides a single connection point.
*   **DB Subnet Group:** An `aws_docdb_subnet_group` is created using two public subnets in different Availability Zones. DocumentDB uses this group to place instances and ensure high availability. *(Note: Using public subnets for a DB subnet group is generally not recommended for production; private subnets are preferred. This configuration is for demonstration simplicity.)*
*   **Storage Encryption:** Data at rest within the DocumentDB cluster is encrypted by default (`storage_encrypted = true`).
*   **EC2 Client with Mongo Shell:** An Amazon Linux 2 EC2 instance is launched and configured using a cloud-init script. This script is templated with the DocumentDB cluster's connection details (cluster endpoint, port, master username, and password) and is intended to:
    1.  Install the MongoDB shell (`mongo`).
    2.  Optionally configure connection aliases or helper scripts.
*   **Secure Credential Management:** A random master password for the DocumentDB cluster is generated by AWS. This password is then retrieved by Terraform and passed to the EC2 client's cloud-init script for setup.
*   **Security Groups:** Network access to the DocumentDB cluster is controlled by a dedicated security group, allowing traffic only from within the VPC.

## AWS Resources Provisioned

*   **VPC (Virtual Private Cloud):**
    *   A new VPC with an Internet Gateway (IGW).
    *   Two **public subnets** created in different Availability Zones (`var.az`, `var.az2`). These subnets are used for the DocumentDB Subnet Group and for hosting the EC2 client.
*   **AWS DocumentDB Cluster (`aws_docdb_cluster`):**
    *   `engine = "docdb"`.
    *   Configurable cluster identifier (`var.docdb_cluster_identifier`).
    *   Configurable master username (`var.docdb_user`).
    *   Configurable port (`var.docdb_port`, default 27017).
    *   A **random master password** is automatically generated by RDS/DocumentDB.
    *   **DB Subnet Group (`aws_docdb_subnet_group`):** Created from the two public subnets.
    *   `storage_encrypted = true` (encryption at rest is enabled).
    *   Configurable backup retention period and preferred backup window.
    *   Configurable preferred maintenance window.
    *   Associated with a dedicated security group (`aws_security_group.demo26_docdb`).
*   **DocumentDB Cluster Instances (`aws_docdb_cluster_instance`):**
    *   A configurable number of instances (`var.docdb_nb_of_instances`) are provisioned and associated with the cluster.
    *   Configurable instance class (`var.docdb_instance_type`).
*   **DocumentDB Security Group (`aws_security_group.demo26_docdb`):**
    *   Allows inbound traffic on the DocumentDB port (e.g., TCP 27017) from within the VPC (specifically, from the EC2 client's security group or VPC CIDR).
*   **Linux EC2 Client Instance:**
    *   An Amazon Linux 2 instance (type `var.al2_inst_type`) launched in one of the public subnets.
    *   An **Elastic IP (EIP)** is associated for a static public IP address.
    *   Uses a **cloud-init script** (`cloud_init_al2_TEMPLATE.sh` via `var.al2_cloud_init_script`) which is templated with:
        *   DocumentDB cluster endpoint.
        *   DocumentDB port.
        *   DocumentDB master username.
        *   The **randomly generated DocumentDB master password**.
        This script sets up the MongoDB client tools (mongo shell).
    *   Associated with its own security group (`aws_default_security_group.demo26_ec2`, typically the VPC's default SG modified or a new one).
*   **EC2 Client Security Group (`aws_default_security_group.demo26_ec2`):**
    *   Allows inbound SSH (TCP port 22) from `authorized_ips`.
    *   Allows all outbound traffic (or at least traffic to the DocumentDB security group on the DocumentDB port).

## Architecture

```
        [ AWS Cloud - Region: var.aws_region ]
                         |
        +---------------------------------------------------+
        |                       VPC                       |
        |                (var.cidr_vpc)                   |
        |                                                 |
        |  +-----------------+   +-----------------+      |
        |  | Public Subnet 1 |   | Public Subnet 2 |      | (In different AZs)
        |  | (var.az)        |   | (var.az2)       |      |
        |  |-----------------|   |-----------------|      |
        |  | EC2 Client Inst |   |                 |      |  DocumentDB Subnet Group
        |  | (EIP, AL2)      |<----TCP/27017------>| DocumentDB Cluster |<--(Spans Subnets)
        |  | - Cloud-Init    |   | (MongoDB compat.)|      |
        |  | - Mongo Shell   |   | - Encrypted     |      |
        |  | (SG: EC2 SG)    |   | - Multiple Inst |      |
        |  |        |        |   | (SG: DocDB SG)  |      |
        |  +--------|--------+   +-----------------+      |
        |           | (SSH)                               |
        |           ▼                                     |
        |     [Internet Gateway]                          |
        +---------------------------------------------------+
                      (Internet)

Traffic Flow:
  - User SSH -> EC2 Client Instance (via IGW & EIP).
  - EC2 Client -> DocumentDB Cluster Endpoint (Private IP, within VPC, TCP port from var.docdb_port).
    - Controlled by EC2 SG outbound rules and DocumentDB SG inbound rules.
```
The DocumentDB cluster and its instances reside within the DB Subnet Group. The EC2 client, in a public subnet for SSH access, communicates with the DocumentDB cluster endpoint using private IP addresses.

## Key Configuration Variables

*   **General AWS & VPC:**
    *   `aws_region`: AWS region (e.g., "us-east-1").
    *   `az`, `az2`: Availability Zones for the public subnets.
    *   `cidr_vpc`: CIDR block for the VPC.
    *   `cidr_subnet1`, `cidr_subnet2`: CIDRs for the public subnets.
    *   `authorized_ips`: IPs/CIDRs for SSH access to the EC2 client.
*   **DocumentDB Specific:**
    *   `docdb_cluster_identifier`: Cluster identifier (e.g., "demo26-docdb").
    *   `docdb_instance_type`: Instance class for DocumentDB instances (e.g., "db.r5.large").
    *   `docdb_nb_of_instances`: Number of instances in the cluster (e.g., 2).
    *   `docdb_user`: Master username (e.g., "docdbadmin").
    *   `docdb_port`: Port for DocumentDB (e.g., 27017).
    *   Variables for backup window, maintenance window, engine version.
*   **EC2 Client Specific:**
    *   `al2_inst_type`: EC2 instance type (e.g., "t3.micro").
    *   `al2_ssh_key_name`: Name of an existing EC2 Key Pair for SSH.
    *   `al2_cloud_init_script`: Path to the cloud-init template file.

## Usage

1.  **Initialize Terraform:**
    ```bash
    terraform init
    ```
2.  **Plan Changes:**
    Review the resources that Terraform will create.
    ```bash
    terraform plan
    ```
3.  **Apply Changes:**
    Provision the AWS resources.
    ```bash
    terraform apply
    ```
    Confirm by typing `yes`.

## Connecting to DocumentDB

After successful deployment:

1.  **SSH into the EC2 Client Instance:**
    Use its Elastic IP (EIP) and your SSH key.
    ```bash
    ssh -i /path/to/your/ssh-key.pem ec2-user@<EIP_EC2_Instance>
    ```

2.  **Use the Mongo Shell to Connect:**
    The cloud-init script (`var.al2_cloud_init_script`) is designed to:
    *   Install the MongoDB shell (`mongo`).
    *   Download the Amazon DocumentDB CA certificate bundle (required for SSL/TLS connections, which DocumentDB enforces).
    *   The RDS connection details (cluster endpoint, port, username, and the **randomly generated password**) are passed as template variables to this script. The script may use these to facilitate connection or provide guidance.

    To connect using the mongo shell:
    ```bash
    mongo --ssl \
          --host <docdb_cluster_endpoint>:<docdb_port> \
          --sslCAFile global-bundle.pem \
          --username <docdb_user> \
          --password <generated_docdb_password>
    ```
    *   `<docdb_cluster_endpoint>`: The cluster endpoint DNS name of the DocumentDB cluster (from Terraform output or AWS console).
    *   `<docdb_port>`: The port DocumentDB is listening on (from `var.docdb_port`).
    *   `<docdb_user>`: The master username (from `var.docdb_user`).
    *   `<generated_docdb_password>`: The **randomly generated password**. You'll need to retrieve this from the Terraform output where it's exposed (e.g., `nonsensitive(random_password.docdb_master_password.result)`). **Note:** Exposing secrets directly in Terraform outputs is generally not recommended for production; consider using AWS Secrets Manager for better secret handling.

    Once connected, you can run MongoDB commands (e.g., `show dbs`, `use your_database`, `db.your_collection.insert({ name: "test" })`).

This setup provides a functional DocumentDB cluster accessible from a pre-configured EC2 client instance. Remember to handle the master password securely, especially in production environments.
